ChaosDB in a nutshell

ChaosDB is a powerful multi-schema, fully-transactional database, which:

  • speaks pathSQL, a SQL dialect with powerful path expressions, ideal for social network applications
  • is written entirely in C++, with a tiny footprint suitable for any device

This document discusses ChaosDB's data model, query languages, interfaces & components, strengths, and additional features.

ChaosDB in a nutshell: data model

  • a very flexible object (aka PIN), similar to javascript objects:
    • compact and flexible representation for structured or semi-structured data
    • equivalent to rdbms rows, oodbms objects, graph db nodes, document db or xml/json documents, rdf triples etc.
  • each property can contain:
    • literals (strings, numbers, dates, blobs etc.)
    • references to other PINs (or sub-elements)
    • heterogeneous collections
    • user-defined sub-structures
  • classes provide fast access to PINs:
    • automatic, synchronous, fully transactional indexing
    • based on user-defined predicates (e.g. CREATE CLASS myclass AS SELECT * WHERE EXISTS(myprop) AND myprop2 IN ('yellow', 'green', 'blue'))

described in more detail in the doc under 'terminology'

ChaosDB in a nutshell: query languages

  • pathSQL extends SQL
    • allows to create, update, delete, undelete and purge objects and classes
    • with extensions for manipulating references and collections
    • the DML is mostly standard, with a few extensions, e.g. very powerful path expressions
  • ChaosDB also provides a low-level C++ query builder (to add languages such as XQuery, SPARQL, LINQ etc.)

ChaosDB in a nutshell: interfaces & components

  • the kernel is self-contained and written entirely in C++ (~1Mb binary size)
  • the kernel exposes C++ interfaces with 3 main entry points:
    • pathSQL
    • protobuf for high-performance in/out streaming
    • direct (embedded) C++ access
  • the server exposes an HTTP interface, with JSON and protobuf support
  • client-side libraries are provided for javascript, python and ruby (planned: java, C++ etc.)

ChaosDB in a nutshell: strengths

  • extremely flexible data model, allowing simultaneous storage and manipulation of table data, objects, graphs, documents etc.
    • optimized for social networks
    • from chaos emerges order
  • coexistence of multiple schema, and data sharing between applications (even unrelated)
  • on-the-fly classification, enabling the above, as well as features like stream processing
  • system portability (OSX, linux, Windows, ARM-based devices)
  • deployment portability (tiny C++ kernel well suited for small devices all the way to cloud servers)
  • ease of use and quick prototyping (reduces or eliminates altogether the need for an ORM)
  • strong commitment to ACID

additional features

  • automatic, synchronous, transaction-integrated free-text indexing
  • ability to insert complex graph structures in a single pathSQL sentence
  • transactional, on-the-fly schema modifications
  • per-object and per-class update notifications
  • multi-user, ACL-driven sharing; optional per-object ACLs
  • ability to access concurrently hundreds of database instances, in one process
  • property and class names represented by URIs, with qname support
  • units of measurement, dimensionality control and unit conversions in calculations
  • strong encryption (optional)
